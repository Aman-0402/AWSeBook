🤝The AWS Shared Responsibility Model

Security in AWS is a shared responsibility between AWS and the customer. Understanding exactly where AWS's responsibility ends and yours begins is one of the most important concepts in cloud security.

AWS Responsibility
"Security OF the Cloud"		 Customer Responsibility
"Security IN the Cloud"
What Protecting the underlying cloud infrastructure Configuring and protecting your workloads on that infrastructure
Examples Physical data centres, hardware, networking, storage hardware, global infrastructure (Regions, AZs, Edge Locations) Customer data, application code, IAM policies, OS configuration, network firewalls, encryption, S3 bucket permissions
Key point You don't have to worry about someone physically accessing AWS servers If you make an S3 bucket public by mistake — that is YOUR responsibility, not AWS's
⚠️ Critical real-world example: Hundreds of data breaches occur every year because customers accidentally make S3 buckets public. AWS's infrastructure is secure — the misconfiguration is the customer's responsibility to fix.

🏛️Seven Security Design Principles

AWS recommends seven principles when designing secure cloud systems:

# Principle What It Means Example
1 Strong Identity Foundation Every user and service must have a verified identity IAM users, roles, and MFA for all access
2 Enable Traceability Track all actions — who accessed what, when, and from where CloudTrail logs every API call; detect suspicious activity
3 Security at All Layers Apply protection at network, application, data, and infrastructure level Encrypt storage + secure APIs + control network + audit code
4 Automate Security Reduce human error with automated checks and alerts AWS Config rules, CloudWatch alarms, security policy automation
5 Protect Data in Transit and at Rest Encrypt data whether it is stored or moving between systems S3 encryption at rest (KMS), HTTPS/TLS for all data transfers
6 Keep People Away from Data Limit direct access to sensitive data — use dashboards instead Analysts view QuickSight dashboards; never touch raw S3 data
7 Prepare for Security Events Have an incident response plan ready before attacks happen Runbooks for data breach, compromised IAM keys, ransomware
Section 02

Access Management and AWS IAM

🔑Authentication vs Authorisation

Concept Question It Answers Example
Authentication Who are you? Username + password + MFA code = identity confirmed
Authorisation What are you allowed to do? Data analyst = can run queries and view dashboards, but cannot delete data or change system settings

🎭Access Control Models

Model How It Works Best For Example
RBAC
Role-Based Access Control		 Permissions assigned to roles; users inherit role permissions Stable organisational structures with clear job functions Admin = full access. Data Engineer = S3 + Glue. Analyst = Athena + QuickSight only.
ABAC
Attribute-Based Access Control		 Permissions depend on attributes (department, location, project tag) Dynamic environments where access depends on context Finance team members can access datasets tagged department=finance
💡 Principle of Least Privilege: Every user and service should have only the minimum permissions required to do their job — nothing more. Start minimal and grant additional permissions only when genuinely needed. Remove permissions when they are no longer required.

🛡️AWS Identity and Access Management (IAM)

IAM is the foundational AWS service for access control. It manages users, groups, roles, and policies across every AWS service.

👤
IAM Users & Groups
Individual identities for people. Group users by function (data-engineers, analysts) and attach permissions to the group — not individual users.
🎭
IAM Roles
Temporary identities assumed by services (EC2, Lambda, Glue). Roles are the AWS-recommended way to grant services access to other services — no hard-coded credentials.
📋
IAM Policies
JSON documents defining allowed or denied actions. Can be attached to users, groups, or roles. Support fine-grained conditions (e.g. allow S3 read but not delete).
🔐
MFA
Multi-Factor Authentication adds a second verification step. Password + time-based mobile code. Essential for all human users, especially those with admin access.
🔗
Federated Identity
Company SSO login → AWS access. Users authenticate with their existing corporate credentials (Active Directory, Okta, Google Workspace) without separate AWS passwords.
🔍
Granular Permissions
Fine-grained control down to individual S3 objects, specific DynamoDB tables, or particular Redshift schemas. User can read one bucket but not delete anything.
Section 03

Data Security, KMS, and Monitoring

🔒Data at Rest vs Data in Transit

Data at Rest Data in Transit
Definition Data stored in databases, S3, data lakes, or any storage system Data moving between systems — application to database, server to server, browser to web server
Encryption method AES-256 encryption managed by AWS KMS HTTPS / TLS 1.2+ for all data movement
AWS implementation S3 SSE-KMS, Redshift encryption, DynamoDB encryption All AWS APIs enforce HTTPS; VPC traffic can use TLS
Example Patient records in S3 bucket — encrypted with hospital's KMS key Credit card data sent from browser to checkout server — encrypted with TLS (HTTPS padlock)

🗝️AWS Key Management Service (KMS)

KMS is the central service for creating, managing, and controlling cryptographic encryption keys across all AWS services.

🔑
Create & Manage Keys
Generate symmetric (AES-256) and asymmetric (RSA, ECC) keys. Set key policies that control who can use or manage each key.
🔄
Automatic Key Rotation
KMS can automatically rotate encryption keys annually — old data remains decryptable, new data uses the fresh key. No manual key management needed.
🖥️
Hardware Security Modules
Keys are stored in FIPS 140-2 validated HSMs — specialised tamper-resistant hardware. Keys never leave AWS KMS unencrypted.
📝
Audit Key Usage
Every key usage is logged in CloudTrail — who decrypted what data, when, and from which service. Essential for compliance.

📋AWS CloudTrail — Traceability

CloudTrail records every API call and user action in your AWS account. It is the primary tool for answering: "Who did what, when, and from where?"

Scenario CloudTrail Captures
Security investigation User X deleted S3 object at 2:14 AM from IP 198.51.100.42
Compliance audit Complete history of who accessed sensitive data and when
Incident response Which IAM user modified the production security group
Change management All CloudFormation stack changes with timestamps and actors

📊Amazon CloudWatch — Monitoring

CloudWatch provides continuous monitoring of AWS resources and applications — collecting metrics, creating alarms, and triggering automated responses.

CloudWatch Feature Example Use Case
Metrics collection CPU utilisation, memory usage, network throughput, Kinesis shard iterator age
Alarms If CPU > 90% for 5 minutes → trigger Auto Scaling + send SNS notification
Log aggregation Collect application logs, VPC flow logs, and Lambda execution logs in one place
Dashboards Real-time operational dashboard showing pipeline health metrics
Anomaly detection ML-powered detection of unusual metric patterns (e.g. sudden traffic spike)
Section 04

Security of Analytics Workloads

Analytics pipelines process large volumes of business-critical data — financial transactions, customer records, medical data, and proprietary reports. AWS recommends three security layers for analytics workloads.

🏷️Layer 1 — Classify and Protect Data

Before securing data, you must know how sensitive it is. Data classification assigns a sensitivity level to every dataset.

Data Type Sensitivity Protection Required Example
Public data Low Basic access control Published company blog posts, public API data
Internal business data Medium Encryption + role-based access Internal reports, website analytics logs
Customer personal data High Strong encryption + audit trails + access logging Names, email addresses, purchase history
Financial / Medical data Very Sensitive Maximum protection + compliance (HIPAA, PCI-DSS) Credit card numbers, patient records, bank statements

Classification metadata is stored in the AWS Glue Data Catalog so every team knows the sensitivity level of every dataset. Data retention policies define how long each classification is kept — for example, financial records for 7 years, session logs for 30 days.

💡 Downstream classification: If a dataset is classified as Sensitive at the source, it must remain classified as Sensitive through every stage of the pipeline — ingestion, storage, processing, and analytics. Security cannot be stripped away mid-pipeline.

🚪Layer 2 — Control Data Access

👑
Data Owner Authority
Each dataset has a designated owner who decides who can read, modify, or delete it. Finance data = Finance team owns it. HR data = HR team controls it.
🪪
User Identity Solutions
Users must authenticate before accessing data. IAM, federated SSO, MFA — all enforced before any data access is granted.
🔐
Authorisation Models
RBAC for stable roles (Engineer, Analyst, Admin). ABAC for dynamic access based on tags, department, or project attribute.
🚨
Emergency Access Process
Pre-defined "break-glass" procedures for system failures or security incidents. Temporary elevated access is granted, fully logged, and revoked immediately after resolution.

🏗️Layer 3 — Control Access to Infrastructure

Risk Solution AWS Tool
Public S3 buckets S3 Block Public Access enabled by default + bucket policies S3 Block Public Access, IAM policies
Open network ports VPC security groups — allow only required ports from required sources VPC, Security Groups, Network ACLs
Unmonitored infrastructure changes Every change to security groups, IAM policies, or databases is logged CloudTrail, AWS Config, CloudWatch
Tampered audit logs Write-once audit log storage — nobody can delete or modify logs S3 Object Lock, CloudTrail log file validation

ML Phase Key Security Action Example Threat / Risk
Phase 1: Identify Goal Validate privacy rules, licensing terms, and legal use of data before starting Using a public dataset that contains personal data without consent
Phase 2: Frame Problem Apply least privilege access — only ML engineers can access training data Analyst accidentally modifies training labels, corrupting the model

Phase 5: Deploy Model Protect against adversarial attacks, restrict API access with IAM authentication Attacker modifies transaction by $1 to bypass fraud detection threshold
Phase 6: Monitor Restrict access to authorised users, monitor for anomalous data access, track model performance degradation Insider threat downloads training data; model accuracy drifts as input data distribution changes

☠️Key ML Security Threats

💉
Data Poisoning
Attackers inject malicious training examples. A fraud model trained on poisoned data learns to classify real fraud as "normal". Prevention: validate sources, monitor anomalies in training data.
🎭
Adversarial Attacks
Carefully crafted inputs designed to fool a deployed model. A $999 transaction instead of $1000 to bypass a fraud threshold. Prevention: input validation, model monitoring, adversarial testing.
🔗
Transfer Learning Risks
Pre-trained models from external sources may contain hidden vulnerabilities or backdoors. Always verify model provenance and test before deploying in production.
💧
Data Leakage
Sensitive data included unnecessarily in training sets. Customer names, addresses, or IDs should be anonymised or removed — ML models only need the features relevant to prediction.
Section 06

Scaling Data Pipelines

Scaling means increasing system capacity to handle more workload. An e-commerce platform normally processing 1,000 users per hour may need to handle 100,000 during a festival sale. The pipeline must scale automatically — without human intervention.

↔️Horizontal vs Vertical Scaling

Horizontal Scaling (Scale Out) Vertical Scaling (Scale Up)
Definition Add more machines to share the workload Increase the resources of an existing machine
Example 1 server → 4 servers, each handling 25% of traffic Upgrade server: 8 GB RAM → 64 GB RAM, 4 CPU → 16 CPU
Limit Virtually unlimited — just keep adding nodes Hardware limit exists — you can't upgrade forever
Fault tolerance High — if one node fails, others continue Low — single point of failure
Cloud preference ✅ Preferred — Kinesis, EMR, DynamoDB all scale horizontally Limited use — good for quick short-term fixes

⚙️AWS Auto Scaling

AWS Auto Scaling automatically adjusts capacity based on real-time demand — adding resources when load is high, removing them when load drops. This optimises both performance and cost.

Scaling Policy How It Works Best For Example Rule
Target Tracking Maintain a specific metric at a target value — scale up or down as needed Steady workloads where you want to maintain consistent performance Keep CPU at 50% — add instances if above, remove if below
Step Scaling Scale in steps based on threshold breaches Workloads with variable but predictable spikes CPU 70% → +1 instance; CPU 85% → +2 instances; CPU 95% → +4 instances
Scheduled Scaling Scale at specific times of day or week Predictable patterns — business hours, end-of-month reporting 8 PM daily → increase capacity; 2 AM → reduce to minimum

🌊Scaling Streaming Pipelines with Kinesis

Kinesis Data Streams uses shards to scale. Each shard provides 1 MB/s write throughput and 2 MB/s read throughput. Adding shards increases pipeline capacity horizontally.

💡 Why buffer with Kinesis before S3? Sending high-velocity event data directly to S3 creates too many small write requests — this is expensive and can cause throttling. Kinesis buffers the stream and Firehose batches it into efficient S3 writes. This improves reliability, scalability, and cost.

⚖️Scaling Trade-offs

Scenario Speed Cost Architecture Choice
Fraud detection (real-time) Milliseconds High (always-on) Streaming: Kinesis + Flink

Weekly analytics Hours Very low (scheduled) Batch: EMR + Spark
Interactive ad-hoc queries Seconds Medium (pay per query) Serverless: Athena
Section 07

Infrastructure as Code and AWS CloudFormation

Traditional infrastructure deployment is manual — engineers click through consoles or run commands to create servers, databases, and networks. This is slow, error-prone, and produces inconsistent environments. Infrastructure as Code (IaC) solves this by defining infrastructure in code files that can be version-controlled, reviewed, and automated.

Manual Infrastructure Infrastructure as Code
Speed Hours or days of clicking through consoles Minutes — run a template, resources appear
Consistency Dev, Test, and Prod environments differ — bugs from inconsistency Identical environments every time from the same template
Change tracking No record of who changed what Git history shows every infrastructure change with author and reason
Disaster recovery Rebuilding from scratch takes days Re-run the template — infrastructure rebuilt in minutes

📝Declarative vs Imperative IaC

Declarative Imperative
Approach Define what the end state should be — the tool figures out how Define how to build it step by step
Example "I want: 2 EC2 instances, 1 S3 bucket, 1 RDS database" "Step 1: create VPC. Step 2: create subnet. Step 3: launch EC2..."
AWS tool AWS CloudFormation, AWS CDK Shell scripts, Ansible, AWS CLI scripts

☁️AWS CloudFormation

CloudFormation is AWS's native IaC service. You define all your infrastructure in a template file (YAML or JSON). CloudFormation creates, updates, and deletes all the resources defined — grouped into a Stack.

Template Section Required? Purpose Example Content
AWSTemplateFormatVersion No Template version declaration 2010-09-09
Description No Human-readable explanation of template "Data pipeline infrastructure for sales analytics"
Parameters No Input values passed at deployment time Environment (dev/prod), instance type, bucket name
Mappings No Static lookup values (region-to-AMI mappings) AMI IDs per region
Conditions No Conditional resource creation Create Multi-AZ RDS only in production
Resources ✅ YES AWS resources to create — the ONLY required section EC2 instances, S3 buckets, RDS databases, IAM roles
Outputs No Values returned after stack creation S3 bucket ARN, load balancer DNS name, database endpoint 
# CloudFormation template — ensure EC2 is created before RDS
Resources:
  WebServer:
    Type: AWS::EC2::Instance
    Properties:
      InstanceType: t3.medium
      ImageId: ami-0abcdef1234567890

  Database:
    Type: AWS::RDS::DBInstance
    DependsOn: WebServer    # RDS waits for EC2 to be ready first
    Properties:
      DBInstanceClass: db.t3.medium
      Engine: mysql
      MasterUsername: admin

🛠️AWS Cloud Development Kit (CDK)

CDK allows developers to define infrastructure using familiar programming languages — Python, TypeScript, Java, C#, or Go — instead of YAML templates. CDK synthesises into CloudFormation templates automatically.

🐍
CDK App → Stack → Constructs
CDK Apps contain Stacks (equivalent to CloudFormation stacks). Stacks contain Constructs (individual AWS resources). Write once in Python, deploy anywhere.
🔁
Reusable Patterns
Package common patterns as reusable CDK constructs — e.g. a "secure S3 data lake" construct that always enables versioning, encryption, and block-public-access.
Module 6 Quiz

🧠 Test Your Knowledge

Some questions require multiple answers — select all that apply, then click Check Answers.

Q 01
A data engineer wants to improve overall data security after detecting suspicious activity. Which principles should they apply?
Select THREE correct answers
✅ Correct three: (1) Protect data in transit and at rest — encryption is a core security principle. (2) Implement a strong identity foundation — IAM and MFA verify every user and service. (3) Enable traceability — CloudTrail and logging let you detect and investigate suspicious activity. Manual processes and single-layer security both contradict AWS security best practices.
Q 02
A system administrator creates cryptographic keys, but as the organisation grows it becomes difficult to audit and manage them. Which AWS service should be used?
✅ AWS KMS is the central service for creating, managing, rotating, and auditing encryption keys. It uses hardware security modules (HSMs), integrates with CloudTrail for key usage logging, and supports automatic annual rotation — solving the management and auditability problem at scale.
Q 03
An administrator needs to identify which IAM user terminated a production EC2 instance. Which AWS service should be used?
✅ AWS CloudTrail logs every API call — including exactly which IAM user (or role) called TerminateInstances, from which IP, at what time. CloudWatch monitors metrics and performance; it does not log identity-level API actions.
Q 04
A data engineer wants to control access to workload infrastructure. Which security best practices should they implement?
Select THREE correct answers
✅ Correct three: (1) Least privilege policies — every user and service gets only the minimum permissions needed. (2) Prevent unintended access — close open ports, disable public S3 access, use private VPCs. (3) Monitor infrastructure changes — CloudTrail and AWS Config track every modification. Public subnets, open SSH, and unrestricted admin access all create security vulnerabilities.
Q 05
A data engineer wants to improve the security of the data processing phase of a machine learning pipeline. Which best practices should they implement?
Select THREE correct answers
✅ Correct three: (1) Keep only relevant data — minimising stored data reduces exposure; unused customer fields shouldn't be in ML training sets. (2) Enforce data lineage — knowing data provenance helps detect misuse or corruption in the pipeline. (3) Protect sensitive data privacy — encrypt and anonymise PII before it enters training pipelines. Resource tagging and instance sizing are operational concerns, not ML security practices.
Q 06
Which statements are TRUE regarding horizontal and vertical scaling?
Select TWO correct answers
✅ Correct two: (1) Adding more EC2 instances = horizontal scaling (scale out) — you distribute the workload across more machines. (2) Upgrading instance type = vertical scaling (scale up) — you make one machine more powerful. Cloud architecture prefers horizontal scaling for its fault tolerance and virtually unlimited ceiling.
Q 07
A data engineer wants to create and deploy infrastructure as code to automate infrastructure deployment. Which AWS service should be used?
✅ AWS CloudFormation is the native IaC service. You define all infrastructure in template files (YAML/JSON), and CloudFormation creates, updates, and deletes resources automatically — enabling repeatable, version-controlled deployments.
Q 08
Which section of an AWS CloudFormation template is the only required section and specifies AWS resources such as EC2 instances or S3 buckets?
✅ The Resources section is the only mandatory section in a CloudFormation template. It defines every AWS resource to be created (EC2, S3, RDS, IAM roles, etc.) along with their properties and dependencies. All other sections — Parameters, Mappings, Conditions, Outputs — are optional.
Q 09
What are reasons a data engineer might use AWS CloudFormation instead of manually setting up AWS resources?
Select THREE correct answers
✅ Correct three: (1) Simplify management — one template defines all infrastructure; no clicking through consoles. (2) Quickly replicate — spin up identical dev, test, and prod environments from the same template in minutes. (3) Control and track changes — templates in Git give full history of who changed what and why. CloudFormation does not auto-create IAM policies or manage costs; CloudWatch handles monitoring.
Q 10
In an AWS CloudFormation template, how can an engineer ensure that an EC2 instance is created before an Amazon RDS database?
✅ DependsOn is the correct CloudFormation attribute. Add DependsOn: WebServer to the RDS resource definition, and CloudFormation will wait for the EC2 instance to reach a CREATE_COMPLETE state before beginning the RDS creation. CloudFormation processes resources in parallel by default — position in the file does not determine creation order.
