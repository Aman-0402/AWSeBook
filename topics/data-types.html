Section 01

The Five Vs of Data

Before building a data pipeline, engineers must understand the characteristics of the data itself. Data is not always the same — it can be huge, fast-moving, messy, or come from dozens of different sources at once.

These characteristics are described using the Five Vs of Data. They directly influence every design decision you make about storage, processing, and infrastructure.

📦
Volume
How much data exists. From megabytes to petabytes — size determines storage and processing architecture.
⚡
Velocity
How fast data is generated and must be processed. Milliseconds (stock prices) to hours (daily reports).
🗂️
Variety
The different types and formats of data — structured tables, semi-structured JSON, unstructured images and video.
✅
Veracity
How accurate and trustworthy data is. Bad data leads to bad decisions — quality must be maintained at every layer.
💎
Value
The useful business insights that can be extracted. The ultimate reason for building any data pipeline.
💡 Why the Five Vs matter: Each V influences a different set of infrastructure choices. Understanding all five before writing a single line of pipeline code saves enormous time and cost later.

❓Key Questions Engineers Ask Before Building

Question Related V Impact on Design
Does the required data exist? Value If not, you must build a collection system first
Where is the data stored? Variety Different formats (CSV, JSON, SQL) need different ingestion approaches
How much data exists and how fast does it grow? Volume Determines whether to use distributed storage (S3, EMR) or a simple database
How fast must data be processed? Velocity Decides between batch processing (hourly/daily) and real-time streaming
What is the data quality? Veracity Determines how much cleaning and validation the pipeline must perform
What are the security requirements? Veracity Healthcare, finance, and PII data require encryption, IAM policies, and audit trails

Volume and Velocity are the two most impactful characteristics on pipeline infrastructure. Together, they drive the expected throughput and scaling requirements of every layer in your pipeline.

🗄️Storage Decisions — Volume & Velocity

After ingestion, where data is stored depends on how much of it there is and how fast it needs to be accessed.

⚙️Processing Decisions — Volume & Velocity

🏦
Big Data Processing
Analysing millions of credit card transactions across the US requires distributed computing. AWS EMR runs Apache Spark across a cluster of nodes — all in parallel.
🚨
Real-Time Fraud Detection
If a card is used in New York then Tokyo two minutes later — the system must flag it instantly. Kinesis + Lambda processes each transaction in milliseconds.

📊Analysis and Visualisation Decisions

Use Case What Users Need AWS Tool
One year of sales history Drill-down from country → state → city → store Amazon QuickSight, Redshift
Factory IoT sensors Machine temperature and error rate updated every few seconds AWS IoT Core + Kinesis + QuickSight
⚠️ Key trade-off: Real-time processing is significantly more expensive than batch. Always ask — does this use case actually require real time, or will hourly batch be sufficient?
Section 03

Variety — Data Types and Formats

Variety refers to the different types and formats of data that pipelines must handle. Modern organisations deal with all three categories — often simultaneously in the same pipeline.

Data Type Structure Query Difficulty Flexibility Examples
Structured Fixed rows and columns Easiest (SQL) Least flexible Database tables, spreadsheets
Semi-Structured Flexible, self-describing schema Medium Flexible JSON, XML, CSV, Parquet
Unstructured No predefined format Hardest (AI/NLP needed) Most flexible Images, video, audio, emails

📋Structured Data

Structured data is organised in rows and columns with a fixed, predefined schema. It is the easiest to query and analyse.

-- Structured data in a relational table
CustomerID | Name  | Purchase | Price
-----------+-------+----------+------
101        | Aman  | Laptop   | 850
102        | Rahul | Phone    | 450

-- Easy to query with standard SQL
SELECT name FROM customers WHERE purchase = 'Laptop';

Example systems: MySQL, PostgreSQL, Oracle, Amazon RDS, Amazon Redshift.

📄Semi-Structured Data

Semi-structured data has some organisation but no strict table schema. It is self-describing — the structure is embedded in the data itself. This is the most common format in modern APIs and web applications.

// JSON — most common semi-structured format
{
  "customer": "Aman",
  "purchase": "Laptop",
  "price": 850,
  "tags": ["electronics", "new"]
}

Common formats: JSON, XML, CSV, Parquet, Avro. Used in web APIs, application logs, and chat data. AWS services: S3 + Glue Data Catalog, Amazon Athena.

🖼️Unstructured Data

Unstructured data has no predefined format and cannot be directly queried with SQL. It requires AI, ML, or NLP to extract meaning. Approximately 80% of all data generated today is unstructured.

📷
Images & Video
Product photos, CCTV footage, medical scans. Requires Amazon Rekognition or CV models to analyse.
📧
Emails & Documents
"The product arrived late and I am unhappy." Requires NLP (Amazon Comprehend) for sentiment analysis.
🎤
Audio & Voice
Call centre recordings, voice commands. Amazon Transcribe converts audio to text for further analysis.
📱
Social Media Posts
Customer opinions, brand mentions, trending topics. Free-form text requiring AI to understand at scale.
💡 Data Temperature: Structured data is sometimes called hot data — frequently used, easy to analyse. Unstructured data is cold data — harder to analyse but more flexible and often contains the most valuable hidden insights.

💡Use Case Examples by Data Type

Data Type Real-World Example Analysis Approach
Structured Query support tickets from January 2024 SQL: SELECT * FROM tickets WHERE date BETWEEN '2024-01-01' AND '2024-01-31'
Semi-Structured Analyse customer chat logs stored as JSON — extract intent and issue type AWS Glue + Athena: parse JSON fields, flatten nested objects
Unstructured Customer support emails: "I am very happy with your product" Amazon Comprehend: Sentiment = Positive (confidence: 97%)

Section 05

Veracity and Value

✅What is Veracity?

Veracity means the accuracy, reliability, and trustworthiness of data. It is not enough to have large amounts of data — that data must actually be correct.

⚠️ Critical rule: Making decisions with bad data is worse than making no decision at all. A wrong business decision caused by incorrect data can cost a company far more than acting on incomplete information.
Data Quality What Happens Real-World Impact
Good Data Reliable insights, accurate analysis Correct business strategy, accurate inventory orders, effective fraud detection
Limited Data Some insights but incomplete understanding Decisions made with partial information — risky but manageable
Bad Data Incorrect conclusions, misleading analysis Wrong treatment in a hospital, incorrect revenue projections, missed fraud alerts

🔎Types of Data Quality Problems

🕰️
Dated Information
Using 2020 COVID statistics in a 2025 analysis. Old data may be irrelevant or actively misleading.
❓
Missing Data
Fields like Age: null or City: (empty) reduce the accuracy of any model or analysis that depends on those fields.
🔀
Ambiguous Values
A field containing "10" — is it dollars, kilograms, or a rating out of 10? Without context, the value is meaningless.
📊
Statistical Bias
A survey conducted only in cities will not represent rural behaviour. Biased data leads to biased conclusions.
🐛
Software Bugs
Incorrect code in a pipeline can corrupt data silently — e.g. a rounding error in price calculations.
✏️
Human Errors
Manual data entry mistakes — entering ₹500 instead of ₹5,000, or typing a wrong customer ID.

🔗Maintaining Veracity Across the Pipeline

Data quality must be actively maintained at every layer — not just fixed once at the beginning.

Pipeline Layer Veracity Action Example
Data Sources Validate source reliability Is this database maintained properly? Is the API trustworthy?
Ingestion Schema validation on arrival Reject records missing required fields; flag outliers immediately
Storage Prevent unauthorised modification AWS S3 versioning, object lock, IAM access policies
Processing Preserve data integrity during joins CustomerID must match exactly across all tables being joined
Analysis Audit trail for data changes Log: "Record 101 modified by user X at 10:30 AM on 2026-03-06"

💎What is Value?

Value is the reason the pipeline exists — the useful business insights that can be extracted from data. Value rests entirely on veracity: if you cannot trust the data, your analysis has no value.

🛍️
Retail Example
Sales data reveals customer preferences, seasonal demand patterns, and which promotions actually drive conversions.
🏥
Healthcare Example
Patient data combined with public health statistics identifies high-risk individuals before a health event occurs.
🏦
Finance Example
Transaction patterns expose fraud, predict credit risk, and identify the most profitable customer segments.

❓Questions Engineers & Scientists Ask About Data Quality

Data Engineer Asks Data Scientist Asks
How is the data source managed and maintained? How was this data collected — survey, sensor, or manual entry?
Who owns this data and who is accountable for its quality? Was a scientific method used — random sampling or biased selection?
How frequently is the data updated? What biases might exist in this dataset?
Do we need all fields, or can we reduce storage by removing unused columns? Is the data complete — are there missing values that would affect the model?
Are audit trails available to track changes over time? How recent is the data — is it still relevant to the business question?
Section 06

Data Cleaning and Best Practices

🧹Common Cleaning Tasks

Problem Example Fix
Missing values Age: null Fill with default, median, or flag for manual review
Duplicate records CustomerID 101 appears twice Deduplicate using primary key
Inconsistent format "USA", "U.S.A", "United States" Standardise to one canonical value: "United States"
Impossible values Age: -5 or Age: 200 Apply validation rules: age must be between 0 and 120
Wrong casing "aman", "AMAN", "Aman" Apply title case or lowercase standardisation 
# Before cleaning — raw record from ingestion
{
  "name": "aman",
  "city": "SURAT",
  "age": null,
  "country": "U.S.A"
}

# After cleaning — ready for analysis
{
  "name": "Aman",
  "city": "Surat",
  "age": 25,
  "country": "United States"
}

📐Best Practices for Data Cleaning

📏
Define "Clean" First
Agree on rules before writing code. Example: age must be between 0 and 120; all phone numbers in E.164 format.
🔍
Trace Errors to Source
Fix root causes, not symptoms. If duplicates keep appearing, investigate the ingestion system — not just the output.
⚠️
Change Data Carefully
A value of 0 might mean zero — or it might mean missing. Never assume. Document every transformation rule.
🗃️
Always Retain Raw Data
Never delete the original. Store both raw and cleaned versions in S3. You may need to reprocess raw data with different rules later.

📊Aggregated vs Immutable Raw Data

Many organisations store only aggregated data — pre-computed summaries. This seems efficient but causes problems later.

Aggregated Data Immutable Raw Events
Storage Smaller — summaries only Larger — every event stored
Flexibility Low — limited to pre-defined summaries High — any question can be answered later
Example Page X: average visit = 75 seconds Every user + timestamp + page + exact duration stored
Limitation Cannot answer: "Which users visit daily?" or "What time has peak activity?" Requires more storage and a well-designed schema
Best practice Store as a derived layer on top of raw This is your source of truth — never overwrite 
# Better approach — store every raw event with timestamp
Timestamp        | User ID | Page   | Time on Page
-----------------+---------+--------+-------------
2024-01-10 01:00 | addi1   | Page X | 30 sec
2024-01-10 04:05 | bett2   | Page X | 120 sec
2024-01-10 06:14 | ct244   | Page Y | 600 sec

# Now you can answer ANY question about this data
# — not just the ones you thought of when you designed the system

Practice What It Means AWS Implementation

✅ Summary — The Five Vs guide every pipeline decision: Volume and Velocity determine your ingestion method and processing architecture. Variety determines your storage format and transformation logic. Veracity determines your cleaning strategy. Value determines whether you should build the pipeline at all.

Q 01
Which data characteristic describes how accurate, precise, and trusted the data is?
✅ Veracity refers to the quality, accuracy, and trustworthiness of data. Volume = size, Velocity = speed, Variety = format diversity, Value = business usefulness.
Q 02
Which statements describe volume and velocity in a data pipeline?
Select THREE correct answers
✅ Correct three: (1) Volume is about how much data — the sheer size of data that must be stored and processed. (2) Velocity is about how quickly data enters and moves — the speed of data flow through the pipeline. (3) Both together drive throughput and scaling requirements — neither alone determines the right architecture. Volume alone does not drive scaling; velocity is not about quantity.
Q 03
Which data type best describes CSV, JSON, and XML files?
✅ Semistructured. CSV, JSON, and XML have a defined format but a flexible schema — not the rigid rows/columns of a relational table (structured), and not completely free-form like an image (unstructured).
Q 04
Which statement about data types is correct?
✅ Structured data (tables, SQL) is the easiest to query but least flexible — it cannot store free-form content. Unstructured data (images, video, text) is the hardest to query (requires AI/NLP) but the most flexible — you can store anything.
Q 05
Which type of data source includes data that is generated continuously by events with a time-based component?
✅ IoT devices, sensors, and events generate continuous time-based data streams — temperature readings every second, website clicks as they happen, GPS coordinates from moving vehicles. This is called time-series data.
Q 06
How could a healthcare company use data to support the most valuable personalized experience for their customers?
✅ Combining multiple data types from multiple sources — internal records, public data, and real-time device data — enables proactive, personalised health alerts. This is the highest-value use of all Five Vs working together.
Q 07
What is an example of structured data?
✅ Relational database tables are the classic example of structured data — organised in rows and columns with a fixed schema, queryable using SQL. JSON/CSV = semistructured. Images = unstructured. Clickstream = semistructured depending on format.
Q 08
Which scenario could benefit from batch ingestion or processing?
✅ Batch processing handles data collected over a period and processed together — overnight report generation is the classic example. Real-time dashboards, fraud detection, and live recommendations all require streaming, not batch.
Q 09
Which scenario describes a challenge to velocity?
✅ Velocity challenge. The system cannot process data fast enough to return real-time recommendations during peak load. Different CSV formats = a Variety problem. Missing lineage = a Veracity problem. Running out of disk space = a Volume problem.
Q 10
Which statement about data veracity is true?
✅ Value rests on veracity. If data is not accurate or trustworthy, the analysis and decisions based on it become unreliable. Veracity must be maintained across every pipeline layer — not just ingestion — and "clean data" has no universal definition; it depends on the use case.
