Section 01

AWS Well-Architected Framework

Before writing a single line of pipeline code, architects need a set of guiding principles. The AWS Well-Architected Framework provides exactly that — a proven set of best practices for building cloud systems that are secure, reliable, and cost-efficient.

💡 Think of it as an architectural checklist. Before launching any AWS data system, run it against the six pillars. Any gaps you find are risks — address them before they become production problems.

🏛️The Six Pillars

⚙️
Operational Excellence
Run and monitor systems to deliver business value. Automate operations so engineers can focus on improvements, not manual tasks.
CloudWatch CloudFormation CodePipeline
🔐
Security
Protect data and systems through encryption, access control, and continuous monitoring. Only authorised users access sensitive data.
IAM AWS KMS AWS WAF Shield
🛡️
Reliability
Systems should continue working even when failures occur. If one server fails, another takes over automatically — no downtime.
Auto Scaling Multi-AZ Backups
🚀
Performance Efficiency
Use computing resources efficiently. Choose the right service for each task — not the most powerful, the most appropriate.
EMR Redshift Lambda
💰
Cost Optimization
Eliminate unnecessary expenses. Pay only for what you use, auto-scale, and delete idle resources. Serverless services like Lambda reduce baseline costs dramatically.
Cost Explorer Spot Instances S3 Lifecycle
🌱
Sustainability
Minimise environmental impact. Efficient cloud systems reduce energy consumption. AWS optimises energy use across its global data centres.
Efficient Design Right-Sizing Auto-Off

🔭Well-Architected Lenses

A Lens extends the core framework with specialised guidance for specific domains. Two lenses are essential for data engineers:

Lens Focus Key Guidance
Data Analytics Lens Analytics pipeline workloads Real-time analytics, big data processing, data lake design, reference architectures
Machine Learning Lens ML training and deployment Data collection, preparation, model training, deployment, and monitoring lifecycle
💡 Why lenses exist: Analytics workloads have fundamentally different requirements from standard applications. A customer-facing app prioritises low latency and few transactions. A data pipeline prioritises high throughput and massive parallel processing. Lenses give you the right advice for the right context.
Application System Data Analytics System
Handles individual transactions (e.g. one order) Processes entire datasets (e.g. all orders this month)
Low latency — milliseconds per request High throughput — terabytes per hour
Many small, fast operations Fewer, large batch or streaming operations
Optimise for user experience Optimise for cost per query and processing speed
Section 02

Evolution of Data Architectures

To understand where modern data architecture is going, it helps to understand where it came from. Data systems have evolved dramatically over five decades.

🕰️Architecture Timeline

Era Architecture Key Characteristic Limitation
1970s Mainframe Systems One central computer, many terminals. Banks stored transaction data here. Extremely expensive. Difficult to scale. Only large organisations could afford it.
1980s Client-Server Desktop applications talking to central database servers. Limited scalability. Server became a bottleneck as users grew.
1990s 3-Tier Internet Presentation → Application → Data layers. E-commerce websites. Still struggled with very large analytical workloads.
2000s+ Cloud Microservices Distributed, independent services. Netflix runs separate recommendation, streaming, and billing services. Complexity of managing many independent services.

📐Big Data Architecture Patterns

🏭
Data Warehouse
Separate analytical database from transactional DB. App DB → ETL → Warehouse → Reports. First mainstream analytics architecture.
⚡
Lambda Architecture
Processes both batch and streaming data simultaneously — a batch layer for accuracy and a speed layer for latency. Results merged at query time.
🌊
Kappa Architecture
Simplification of Lambda — everything is treated as a stream. One processing layer handles both real-time and historical data reprocessing.
☁️
Modern Cloud Architecture
Combines data lake + warehouse + processing + ML into one unified, governed platform on AWS. The architecture covered in this module.

📚The Catalog Layer

Alongside storage, organisations need a metadata catalog — a registry of what data exists, where it is, and what its schema looks like. Without a catalog, engineers spend hours searching for datasets.

📖
AWS Glue Data Catalog
Central metadata repository. Stores dataset name, schema, S3 location, owner, and update frequency. Shared across Athena, Redshift Spectrum, and EMR.
🤖
AWS Glue Crawlers
Automatically scan S3 buckets and databases to detect schema. No manual schema definition needed — crawlers populate the catalog for you.
🔍
Redshift Spectrum
Query S3 data directly using Redshift SQL — without moving data into the warehouse. Reads catalog metadata to understand the S3 file structure.
-- Redshift Spectrum: query S3 data without loading it
SELECT
    region,
    SUM(sales_amount) AS total_sales
FROM s3_spectrum.sales_data   -- data lives in S3, not Redshift
WHERE year = 2025
GROUP BY region;
Section 06

The Processing Layer

Raw data is rarely ready for analysis. The processing layer transforms data from its raw state into curated, analysis-ready datasets — cleaning, reshaping, joining, and aggregating it.

Raw Data Problem Processing Solution Example
Missing values Fill with default or flag for review age: null → age: 0 (or exclude from analysis)
Duplicate records Deduplicate on primary key CustomerID 101 appearing twice → keep one
Inconsistent casing Standardise to title case "aman", "AMAN" → "Aman"
Wrong data types Cast to correct type "850" (string) → 850 (integer)
Non-normalised values Map to canonical form "US", "U.S.A", "United States" → "United States"

🛠️Processing Services by Use Case

🔀
SQL-Based ELT
Load data first, then transform using SQL inside Redshift. Best for structured data with clear schemas. Amazon Redshift handles massive analytical transformations.
🐘
Big Data (EMR)
Distributed Spark and Hadoop on Amazon EMR. For datasets too large to fit on one machine — process billions of rows in parallel across a cluster.
⚙️
Near Real-Time ETL
AWS Glue for serverless ETL jobs. Processes data quickly without managing servers — crawl schema, transform, and write to Redshift or S3 automatically.
⚡
Stream Processing
Amazon Managed Service for Apache Flink, or Spark Streaming on EMR. For continuously arriving data requiring real-time aggregation, filtering, or alerting.
# AWS Glue PySpark ETL job — clean and transform sales data
from awsglue.transforms import *
from pyspark.sql.functions import col, initcap, trim

# Load raw data from S3 via Glue Data Catalog
datasource = glueContext.create_dynamic_frame.from_catalog(
    database="raw_db",
    table_name="sales"
)

df = datasource.toDF() \
    .filter(col("amount") > 0) \               # Remove zero/negative amounts
    .dropDuplicates(["order_id"]) \          # Remove duplicates
    .withColumn("city", initcap(trim(col("city"))))  # Standardise casing

# Write curated output to S3 in Parquet format
df.write.mode("overwrite").parquet("s3://curated/sales/")

🔍Amazon Athena — Interactive SQL on S3

Athena lets analysts run SQL directly against files in S3 — no database server, no loading data. It reads schema from the AWS Glue Data Catalog and charges per query scanned.

-- Athena query: total revenue by region for 2025
SELECT
    region,
    SUM(revenue)     AS total_revenue,
    COUNT(order_id)  AS total_orders
FROM curated.sales_2025    -- reads directly from S3 Parquet files
GROUP BY region
ORDER BY total_revenue DESC;

Streaming Analytics Pipelines

Some business problems cannot wait for the next batch job. When milliseconds matter — fraud detection, live IoT monitoring, real-time recommendations — you need a streaming analytics pipeline.

Batch Pipeline Streaming Pipeline
Data collected and processed on a schedule Data processed continuously as it arrives
Results available after the batch completes Results available within seconds or milliseconds
Lower cost — infrastructure runs only when scheduled Higher cost — infrastructure runs 24/7
Example: overnight sales report Example: real-time fraud detection

🔑Core Streaming Services

🌊
Amazon Kinesis Data Streams
Ingests continuous data from producers at any scale. Data is retained for up to 365 days — consumers can replay the stream from any point.
🚿
Amazon Kinesis Firehose
Fully managed delivery stream — automatically buffers, compresses, and loads data into S3, Redshift, or OpenSearch. Zero pipeline code required.
⚡
Apache Flink (Managed)
AWS-managed real-time stream processing. Run stateful computations like windowed aggregations, joins across streams, and pattern detection.
🔍
Amazon OpenSearch
Real-time search and dashboards on streaming data. Ideal for log monitoring and operational analytics where data arrives continuously.

💳Full Example — Real-Time Fraud Detection

# Apache Flink rule — flag suspicious transaction pattern
# Trigger: same card used in two different countries within 5 minutes

SELECT
    card_id,
    FIRST(country)   AS country_1,
    LAST(country)    AS country_2,
    DATEDIFF(minute, FIRST(event_time), LAST(event_time)) AS minutes_apart
FROM transactions
GROUP BY card_id, TUMBLE(event_time, INTERVAL '5' MINUTE)
HAVING FIRST(country) <> LAST(country);   -- different countries = alert

Q 01
Which statement describes how data architectures evolved from 1970 to the present?
✅ Data stores evolved to adapt to increasing demands of data volume, variety, and velocity. Modern data architectures evolved to handle the 3Vs of big data — Volume, Variety, and Velocity. As data generated by businesses, users, and internet services increased rapidly, data storage and processing systems evolved to manage these growing demands efficiently.
Q 02
Which statement describes a key design approach for modern data architecture in the AWS Well-Architected Data Analytics Lens?
✅ Store data in a centralized data lake so it can be accessed by analytics tools and AI/ML workloads. Modern architectures use a centralized data lake to store large volumes of structured and unstructured data.

Q 04

✅ Modern data architecture supports both inside-out and outside-in data movement, and also movement between different purpose-built data stores.

What is the role of AWS Glue in the AWS modern data architecture?
✅ AWS Glue is a serverless ETL service used to extract, transform, and load data between different data sources.
